## Context

The vault generates path components in two places: effort folder slugs (from Claim
titles) and Treasury note stems (from the refine executor). Both must be safe across
Windows (reserved device names, forbidden characters), macOS (NFC normalisation,
colon-in-filenames), Linux (no such restriction, but cross-platform compat needed),
and Obsidian's wikilink parser (which treats `#`, `|`, `[`, `]` as syntax).

## Goals / Non-Goals

**Goals:**
- One authoritative Python module (`vault_naming.py`) declaring all rules as data
- `--check NAME` flag for use by the pre-commit hook (exit 1 on violation)
- Language-neutral JSON mirror for non-Python consumers
- Enforcement at exactly two hard boundaries (executor + hook)

**Non-Goals:**
- Sanitizing / auto-correcting non-conforming names (reject, never silently fix)
- Enforcing on pre-existing files (new and renamed files only; existing names grandfathered)
- Runtime path-component extraction from arbitrary user input (caller's responsibility)

## Decisions

**Single module, declarative RULES dict.** The rules are data, not logic. The
`RULES` dict is the source of truth; `validate_name()` interprets it. The JSON
mirror is generated by running the module — no separate maintenance.

**Two-boundary enforcement.** The hook fires on every commit (including agent and
sync tool commits), making it the backstop. The executor adds a second check
specifically for machine-generated `target_note` stems. Neither trusts the producer.

**Grandfathering.** The hook filters `--diff-filter=AR` (Added or Renamed only).
Existing non-conforming names in the repo are not broken by adding the hook.

## Risks / Trade-offs

- The hook requires `VAULT_ROOT` to be set in the shell environment; if missing,
  the hook fails closed (safe: blocks commit with an informative error).
- `slugify()` in the module is best-effort (ASCII output); callers that need
  non-ASCII slugs must pre-process; out of scope for this change.
